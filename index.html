前言

工具世界正在劇烈震盪

本章帶我們跳進那個變化萬千、令人眼花繚亂的 vibe coding 工具世界。如果你對以前那個穩定、可以長期使用的 IDE 感到懷念，你並不孤單。作者將在這場「寒武紀大爆發」中為我們導航。

章節的核心圍繞三大問題：何時使用 AI 代理？何時使用聊天助理？IDE 又在這一切中扮演什麼角色？此外也深入介紹了 MCP（模型上下文協定），一個被比作「AI 助理在職訓練與遠端控制系統」的新標準。

作者親身經歷
30 年前

Steve 開始每天使用 Emacs，工具鏈穩定如磐石。

15 年前

IntelliJ 花了 5 年時間才佔領 Steve 工作流程的 50%（取代 Emacs）。

2025 年初

Steve 在一週內就幾乎不再使用 IDE 和 Emacs——AI 工具的崛起速度令人咋舌。

2025 年中

Emacs 又回來了，但不再用於寫程式，而是用來協調 AI 代理——這個轉變本身就說明了一切。

你的本事不在於你的編輯器、你的 shell，或是你的代理框架。你真正的資產，是你在一個又一個專案之中累積的多年經驗和辛苦磨練出的直覺。

13-1

開發工具的寒武紀大爆發

過去選擇 IntelliJ 或 VS Code，就像買房子——你知道自己會在裡面住好幾年，甚至幾十年。那樣的日子已經過去了。就像一顆隕石砸中開發者工具鏈，觸發了突然又混亂的寒武紀大爆發，數百個 AI 工具幾乎是在一夜之間出現（又消失）。

IDE、代理、還是命令列？

想要解決一個問題時，會面臨令人眼花繚亂的選擇：聊天機器人、整合進 IDE 的程式助理、獨立 AI 代理、遠端代理，還有可靠的終端機……以下是本書的判斷方式。

🤖
AI 代理

適合自動化執行的複雜任務，可以自主持續迭代至完成。

💬
聊天型助理

快速診斷、搜索或生成樣板程式碼，也適合深夜腦力激盪。

🛠
IDE

複雜的靜態分析、語意索引與大規模確定性重構，無可取代。

⬛
命令列

Anthropic 開發者的新選擇，一旦親自用過就能立刻明白原因。

IDE 的深厚優勢

IDE 是工程界的傳奇，投入了數千人年的工程設計。它透過 LSP（Language Server Protocol）對程式碼庫進行複雜的專有分析，提供深度語意索引。在 vibe coding 的世界，IDE 的人類使用量會下降，但這些索引能力依然至關重要——它們幫助 AI 在程式碼中找到方向，就像幫助人類一樣。

更重要的是，與 AI 不同，IDE 的工具具有確定性。使用 IDE 或大型重構工具進行重構，幾乎都比 AI 自行嘗試相同重構更簡單、成本更低且更精確。一些 IDE（例如 IntelliJ）現在也提供 MCP 伺服器，讓 AI 代理可以直接存取這些強大功能。

13-2

模型上下文協定（MCP）：
連結 AI 與工具

MCP 是 Anthropic 提出的模型上下文協定（Model Context Protocol）。書中用了一個精彩比喻：你用了幾十年的那台客製化製麵機，對新來的副主廚是個問題——他需要專門訓練才能操作那台老古董。同樣的，AI 也需要一套標準方式來存取工具和服務。MCP 就是這套標準，是 AI 助理的在職訓練與遠端控制系統。

MCP 可能是目前世界上最重要的新網際網路協定。MCP 很可能會成為新的 HTTP，因為這是將一切連接到 AI 的關鍵，是現在已知最受支援的讓 AI 存取工具與資料的方式。

MCP 是什麼？

MCP 讓 AI 代理能夠調用工具和服務、與外部系統互動，藉此取得即時資訊。它讓 AI 能夠把任何來源的資料或工具的輸出納入上下文窗口，改善上下文的內容。這讓 AI 更加接近一個全面的、像人類一樣的助手，也代表 AI 能執行工作流程中自訂的、需要互動的，還有高度視覺化的部分。

實戰案例：讓 AI 操作遊戲 UI

Steve 正在開發遊戲《Wyvern》，想讓 AI 為遊戲建立一個單一、現代化的 Node/React 用戶端，目標是取代 5 個老化的原生用戶端。為了讓 AI 代理更有效率，他安裝了一個 MCP 伺服器連接到 Puppeteer（網頁瀏覽器互動自動化工具）。

MCP 互動流程示意
AI 助手 ──▶ MCP ──▶ click(selector: "#login-button")
MCP ──▶ AI 助手 ──▶ ✅ Clicked
AI 助手 ──▶ MCP ──▶ getText(selector: ".welcome")
MCP ──▶ AI 助手 ──▶ ✅ "Welcome back, Adventurer!"

開啟 MCP/Puppeteer 之後的差異可謂天壤之別，就像打開燈讓 AI 能夠「看見」一樣。

13-3

MCP 技術實作：魔法背後的機制

底層原理

MCP 的核心是一個遠端程序呼叫（RPC），可能是 JSON-RPC 2.0 over HTTP 或 WebSocket，都非常常見。其中有三個會動的部分：

  1. MCP 用戶端：AI 助理
  2. MCP 伺服器端：轉接器
  3. 工具或服務：廚房裡的客製製麵機
// from AI → MCP server
{
  "jsonrpc": "2.0",
  "id": 42,  // request-id，可使用非同步或平行的 RPC
  "method": "tools/call",
  "params": { "name": "fetch_weather",
              "arguments": { "location": "San Francisco" } }
}

// MCP server → AI（回覆）
{
  "jsonrpc": "2.0",
  "id": 42,  // response-id
  "result": { "ok": true }
}

這兩則訊息——請求和回覆——就構成了你提供給 AI 的「詞彙」。所有高層級的操作，例如登入、填寫表單或解析結果，都是由這些基礎組合而成。

MCP 快速啟用

https://modelcontextprotocol.io/docs/develop/build-server

MCP 的設計宗旨就是單純。最好的入門方式是依照 ModelContextProtocol.io 上的快速入門範例來操作。令人意外的是，整個範例只有 4 個小型 Python 函式：一個負責呼叫天氣服務、一個生成天氣報告、一個處理天氣警報、最後一個處理輸出的格式。建好 MCP 伺服器後，任何支援 MCP 的用戶端都可以在設定裡加入這個伺服器。

13-4 · 小結

在工具爆發時代的生存之道

你看到了開發者的工具世界，從過去穩定的岩床轉變成激流般的快速流動。Steve 在短短一週內放棄了累積 30 年的 Emacs 肌肉記憶，又在幾個月後轉了個髮夾彎，讓 Emacs 成為 AI 代理的協作核心。

最重要的是，真正的力量不在於任何特定的工具，而是要培養一種能力，可以適應並整合所有能夠達成你需求的工具。

核心原則
  • 你的編輯器不能代表你： 最珍貴的資產是經年累月獲得的經驗和辛苦培養出的直覺，而不是這個月正流行的開發工具。
  • 依據情境選擇適合的工具： AI 代理適合自動化執行、聊天型助理適合快速分析、IDE 適合複雜的解析、聊天機器人適合腦力激盪。
  • 擁抱 MCP、讓 MCP 成為萬用轉接器： MCP 能讓 AI 從學識淵博的顧問轉變為能幹的現場助手。
  • 與同僚保持交流： 時常和其他喜歡嘗試新工具的開發者交流心得。
  • 預留後路： 掌握多種工具，在主要工具遇到阻礙時仍有退路。
  • 藉助 AI 的協助學習新工具： AI 助理已經瞭解所有文件，不需要浪費時間自己研究。

下一章會探索 3 個開發迴圈中的第 1 個——建立穩固的認證與驗證程序，確保 AI 副主廚確實做出你指定的料理。
